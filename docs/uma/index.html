Universal Microservices Architecture

A portable, contract-driven execution model for distributed systems. Business logic that runs where it makes sense - browser, edge, cloud, AI pipeline - without rewriting it every time.

What is UMA?

Universal Microservices Architecture is an execution model that separates business logic from runtime environment. The central insight is simple: the environment a piece of logic runs in is a deployment decision, not an architectural one. UMA makes that separation explicit and enforces it through contracts.

In a traditional architecture, the same business rule gets written multiple times. A pricing formula lives in the backend. A version of it gets duplicated in the browser for instant feedback. A third version runs at the edge for performance. A fourth gets rewritten when the AI pipeline needs it. Nobody planned it that way. It happened because the architecture had no mechanism to prevent it.

UMA introduces that mechanism. Each business capability is packaged as a Universal Microservice: a self-describing, contract-governed unit that compiles to WebAssembly and runs identically across any compliant runtime. The contract defines what the service does, what it depends on, and what it guarantees. The implementation is compiled once. The deployment target is a configuration, not a rewrite.

The architecture model

A UMA system has three layers. The capability layer contains the Universal Microservices - the portable business logic compiled to WASM. The contract layer governs how services discover each other, declare dependencies, and validate behavior. The runtime layer provides the execution environment: browser, edge worker, cloud function, or AI pipeline.

The contract is the interface between layers. It is machine-readable, version-aware, and lifecycle-tracked. A service that satisfies a contract is interchangeable with any other service that satisfies the same contract, regardless of the runtime it runs in. This is what makes composition possible at scale without coupling teams to each other's implementation decisions.

The capability registry is where services register and discover each other at runtime. It is not a service mesh or an API gateway. It is a coordination layer that matches capability requests to available implementations based on contract signatures. A service asks for a capability by contract, not by name. The registry resolves the match. The runtime executes the result.

Core Principles

Contract-first

Every service defines its interface, dependencies, and guarantees in a machine-readable contract before implementation begins. The contract is the source of truth - not the code. This forces architectural decisions to happen before implementation, which is the only order in which they are cheap to make.

Runtime-agnostic

Business logic compiled to WebAssembly runs unchanged across browser, Node, Deno, Cloudflare Workers, and AI pipelines. The execution environment is a deployment decision, not an architectural one. Adding a new runtime does not require touching the business logic.

Composable by design

Services are discovered, validated, and composed at runtime through a capability registry. Systems built with UMA can adapt to new requirements without structural rewrites. Adding a new capability means registering a new service, not modifying existing ones.

AI-navigable

Every service has declared intent, explicit dependencies, and a lifecycle-aware contract. AI agents can reason about what a system does, why each part exists, and how to extend it safely - without reading implementation code or relying on documentation that may be stale.

Behavioral coherence

When the same business logic runs across multiple environments from a single implementation, drift becomes structurally impossible. There is one version of the truth. Changes propagate everywhere the capability is deployed. Inconsistencies between environments are detected at contract validation, not discovered in production.

How it differs from microservices

Traditional microservices partition a system by deployment unit. Each service is a process with a network boundary. UMA partitions a system by business capability with a contract boundary. The difference matters at scale.

A microservices architecture tells you where code runs. A UMA architecture tells you what code does, why it exists, and what it guarantees - and lets the runtime answer the where question independently. The result is a system where the same service can run as a browser module, an edge function, and a cloud service without changing a line of implementation code.

UMA is not a replacement for microservices. It is a layer above them. The runtime adapters that execute UMA capabilities can be microservices. What changes is that the business logic inside those services is portable, contract-governed, and environment-independent.

The problem it solves

The same business logic gets rewritten three, four, five times across browser, backend, edge, and AI pipelines. Most architectures assume the runtime too early and teams pay that cost forever - in maintenance, in drift, in onboarding time, and in the inability to adapt when a new execution environment arrives.

The UMA book documents thirteen specific patterns where this cost appears, with runnable Rust and WASM implementations for each. The companion repository enforces 100% business logic coverage in CI, which means every claim in the book has a test that proves it. The reference application at universalmicroservices.com runs the same business logic across browser and edge from a single WASM build.
