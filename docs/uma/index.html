Universal Microservices Architecture

A portable, contract-driven execution model for distributed systems. Business logic that runs where it makes sense — browser, edge, cloud, AI pipeline — without rewriting it every time.

What is UMA?

UMA separates business logic from runtime environment. Instead of rewriting the same logic for browser, backend, edge, and AI pipelines, you write it once as a portable, self-contained unit — a Universal Microservice — and deploy it wherever it makes sense.

Each service is governed by a contract that defines what it does, what it needs, and what it guarantees. The contract is the interface. The implementation can run in any compliant runtime — Rust, WASM, browser, edge worker, or AI pipeline.

Core Principles

Contract-first

Every service defines its interface, dependencies, and guarantees in a machine-readable contract before implementation begins. The contract is the source of truth — not the code.

Runtime-agnostic

Business logic compiled to WebAssembly runs unchanged across browser, Node, Deno, Cloudflare Workers, and AI pipelines. The execution environment is a deployment decision, not an architectural one.

Composable by design

Services are discovered, validated, and composed at runtime through a capability registry. Systems built with UMA can adapt to new requirements without structural rewrites.

AI-navigable

Every service has declared intent, explicit dependencies, and a lifecycle-aware contract. AI agents can reason about what a system does, why each part exists, and how to extend it safely.

The problem it solves

The same business logic gets rewritten three, four, five times across browser, backend, edge, and AI pipelines. Most architectures assume the runtime too early and teams pay that cost forever — in maintenance, in drift, in onboarding time, and in the inability to adapt. UMA eliminates the assumption. Logic and runtime are separate concerns from the start.
