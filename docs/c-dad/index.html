Contract-Driven AI Development

The missing foundation in AI-assisted software development. AI tools can describe what code does - but not why. Contracts fill that gap.

What is C-DAD?

Contract-Driven AI Development (C-DAD) is a framework where verifiable contracts - not prompts, not comments, not documentation - define how humans, AI systems, and runtimes collaborate to build software.

The difference between a specification and a contract is verifiability. A specification describes intent. A contract enforces it. C-DAD makes every piece of software behaviour a first-class, machine-readable, lifecycle-aware contract.

Core Principles

Contracts over code

AI tools automate structure. They cannot automate understanding. Contracts declare why software behaves as it does - the rationale, the constraints, the governance rules. This is the layer AI tools need to reason about systems, not just describe them.

Verifiable, not just readable

Contracts in C-DAD are immutable units of record with explicit lifecycle states: Draft, Active, Deprecated, Retired. They carry provenance, validation results, and audit trails. They can be signed, distributed, and checked - not just read.

Hybrid governance

Not everything should be automated. C-DAD defines explicit handoff points where human judgment is required - and makes those points visible in the contract itself. Automation handles the routine. Humans govern the consequential.

Machine-first authoring

Contracts are authored as machine-readable manifests with human-friendly documentation generated from them - not the other way around. The machine-readable layer is the source of truth. The narrative is derived.

The problem it solves

AI coding tools can write code that looks right. They cannot know if it is right - because right is determined by intent, and intent was never declared. C-DAD makes intent a first-class citizen of the development process. Every behaviour has a contract. Every contract has a reason. Every reason is machine-readable.
